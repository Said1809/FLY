## 2. Бизнес-контекст

**Проблема.** Студенты и пользователи, ведущие учебную и личную деятельность,
нуждаются в простом инструменте планирования задач с привязкой к дате и времени.
Разрозненные заметки, бумажные планеры и универсальные таск-менеджеры часто
перегружены функциями, требуют постоянного интернета или не дают наглядного
недельного и месячного обзора расписания.

**Решение.** Кроссплатформенное клиентское приложение Fly: создание задач с датой
и временем, недельная и месячная навигация по календарю, отметка выполнения,
персонализация интерфейса. Данные задач хранятся локально (SQLite) и доступны
офлайн; учётная запись пользователя обеспечивается облачной аутентификацией Supabase.

## 3. Цели проекта

1. Предоставить пользователю удобный инструмент планирования задач с календарной
   навигацией (неделя и месяц).
2. Обеспечить локальное хранение данных и работу без постоянного подключения к сети.
3. Реализовать персонализацию интерфейса (тема, шрифт, акцентный цвет).
4. Продемонстрировать применение многослойной архитектуры и паттерна Repository
   в кроссплатформенной разработке на Flutter.

## 4. Ключевые показатели успеха

| Показатель | Целевое значение |
|------------|------------------|
| Число экранов приложения | ≥ 5 |
| CRUD-операции над задачами | Полный цикл (создание, чтение, обновление, удаление) |
| Платформы сборки | Android APK, Web, Windows |
| Покрытие кода тестами | > 40% (целевой показатель методички) |
| Время отклика UI при переключении даты | < 200 мс |

## 5. Границы проекта (Scope)

**В составе MVP:** регистрация и вход (Supabase Auth), CRUD задач, недельный
календарь, месячный календарь с индикацией статуса дней, отметка выполнения,
настройки темы/шрифта/цвета, локальная БД SQLite.

**Вне MVP:** синхронизация задач между устройствами, совместные списки, push-уведомления,
публикация в магазинах приложений, веб-админка.

## 6. Ключевые риски

| Риск | Стратегия реагирования |
|------|------------------------|
| Потеря данных при переустановке | Документировать локальную природу хранения; в перспективе — облачная синхронизация |
| Зависимость от Supabase для входа | Кэширование сессии; вход только при первом запуске |
| Сложность кроссплатформенной сборки | Приоритет Android; проверка через `flutter doctor` |
| Нарушение слоёной архитектуры | Code review, разделение по каталогам `presentation`, `controller`, `model` |
